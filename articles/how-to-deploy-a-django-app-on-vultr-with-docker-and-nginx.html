Step 2 — Core implementation

Start with the smallest possible working version.[1] Resist the temptation to add features before the core works.

Key principles to follow during implementation:

  • Write code that is easy to delete, not just easy to extend.
  • Use explicit names — clarity beats cleverness every time.

  • Read error messages carefully — they almost always tell you exactly what is wrong.

Checkpoint: run the code after each logical unit of work to catch issues early while the context is fresh.

Step 3 — Testing your implementation

Do not skip this section. Untested code is broken code you have not found yet.[3]

Minimum verification steps:

  1. Happy path — the expected inputs produce the expected outputs.[1] Run through the most common use case end-to-end and confirm the result matches your expectations exactly.
  2. Edge cases — empty inputs, maximum sizes, unexpected types. These are where most production bugs hide.[2] Test with an empty string, a very large input, and at least one input that should trigger an error.
  3. Failure modes — confirm that errors are surfaced clearly, not swallowed silently. Disconnect from the network, provide invalid credentials, or pass malformed data. The error messages should tell you exactly what went wrong and where.
  4. Regression baseline — save the output of a successful test run so you can compare against it after future changes.[3] This is especially important for output formats like JSON or HTML where subtle changes can break downstream consumers.

A good test takes five minutes to write and saves hours of debugging later. If you are short on time, at least run the happy path manually and capture the output so you have a baseline to compare against.

Common errors and how to fix them

Error: "command not found" or "module not found"

The tool or package is not on your PATH.[1] Confirm the install succeeded and that you have restarted your shell or sourced your profile after installation.

Error: permission denied

You are trying to write to a location owned by another user or by root.[2] Use a local install path or adjust permissions on your project directory — do not reach for sudo as a first response.

Error: unexpected token / syntax error

Check the language version your runtime is using.[3] New syntax may not be supported in older runtimes. Confirm with --version.

Error: works on my machine, fails in CI

Your local environment has something the CI environment does not.[1] Common culprits: environment variables, system packages, or implicit dependency versions. Lock your dependencies explicitly.

Going further: production considerations

The tutorial above gives you a working foundation.[2] Before deploying to production, consider these additional steps:

  • Error handling — wrap external calls and I/O operations in proper error handling.[3] Log failures with enough context to debug without reproducing the issue locally.
  • Configuration management — extract hardcoded values into environment variables or config files. Twelve-Factor App principles (12factor.net) are a solid guide here.[1]
  • Monitoring — add health checks, structured logging, and basic metrics from day one.[2] You will need them the first time something breaks in production, and adding observability after the fact is always harder than building it in.
  • Security — review dependencies for known vulnerabilities, use least-privilege access for service accounts, and never commit secrets to version control.[3]
  • Documentation — write a README that explains how to set up, run, and deploy the project. Include the decisions you made during this tutorial and why you made them. Future contributors (including your future self) will thank you.
